## 二、数据流

```text
原始文件（只读）          KMFA_MetaData/
    ↓ 只读扫描 + hash
raw manifest             local_runtime/metadata/
    ↓ 抽取
raw_extracted_value      带 source_file / sheet / cell / page
    ↓ 标准化（整数分，禁浮点）
staging_structured_row
    ↓ 身份匹配（≥0.85 自动，0.70–0.85 人工，<0.50 拒绝）
canonical_fact
    ↓ 计算
derived_metric           收入、成本、毛利、毛利率、未归集成本池
    ↓
report_reference → dashboard_display
```

**每一跳都必须留 lineage。** 断一跳，报告等级降到 `D`。

## 三、业务逻辑

> 系统**怎么实现**规则。规则本身在 `03_口径字典.md`。

| 环节 | 实现要点 |
|---|---|
| 金额 | 整数分或 `Decimal`，禁止浮点数。0.01 元差异必须进差异队列，不得静默通过 |
| 身份匹配 | 8 维加权：合同号 .20、项目名 .18、对手方 .15、主体 .10、日期 .12、金额特征 .12、负责人 .06、源 hash .07 |
| 差异 | 任意不符 → 待处理队列 → 人工裁定写裁定事件 → 后端重跑派生链 |
| 原始数据 | 只读。前端不得改 raw、raw hash、raw 抽取值 |

## 四、配置参数（设计意图）

> **当前值和怎么改，在 `06_运维手册.md`。这里只说为什么这么设。**
> 两处由 `machine/config.yaml` 同一份事实渲染，不会打架。

| 参数 | 为什么是这个值 |
|---|---|
| `zero_delta_cents: 0` | 项目成本是对外口径，一分差异就是错，没有容忍带 |
| `failure_threshold_cents: 1` | 1 分即失败，比 0 大是为了让 0 有意义（0 表示"必须完全相等"） |
| `strong_auto_match: 0.85` | 低于此值自动匹配会把不同项目并成一个，代价远大于人工复核 |
| `human_review: 0.70` | 0.70–0.85 是"像但不确定"，交人 |
| `reject_below: 0.50` | 低于 0.50 连候选都不算，避免污染队列 |
| `float_allowed: false` | 浮点在累加万级行时必然产生分级误差，与零差异互斥 |
