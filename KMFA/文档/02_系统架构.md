## 四、旧业务数据从哪到哪

原始文件（只读）→ 只读扫描并计算摘要 → 原始清单 → 抽取（带源文件、表、单元格、页）→ 标准化（整数分，禁浮点）→ 身份匹配（不小于 0.85 自动，0.70 到 0.85 人工，低于 0.50 拒绝）→ 规范事实 → 计算派生指标（收入、成本、毛利、毛利率、未归集成本池）→ 报告引用 → 看板展示。每一跳都必须留血缘；断一跳，报告等级降到 D。源优先级链共 7 环：原始授权导出 → 原始抽取值 → 暂存结构化行 → 规范事实 → 派生指标 → 报告引用 → 看板展示；第 1 环从未接通，导致第 2 到 7 环全部悬空。

## 五、旧业务关键参数为什么这么定

当前值和改哪里在 `06_运维手册.md`，这里只讲**为什么是这个值**。

| 参数 | 为什么定这个值 |
|---|---|
| GitHub 源码仓库 | KMFA 已从旧主仓拆出；唯一公开源码归属必须明确，否则执行者会把修复写回禁推送的历史检出目录。 |
| 稳定本机入口 | 自动化和已安装技能链接需要一个不随日期任务目录变化的检出目录；其他机器可使用任意克隆路径并从仓库根发现。 |
| 清理快照根 | 真实运行数据不进公开仓库；清理前以逐字节一致性通过的私有快照保留 DWS、考勤、日常检查和资金周报状态。 |
| 零差异分 | 项目成本是对外口径，一分差异就是错，没有容忍带。调大等于放弃零差异，报告等级封顶 B。 |
| 失败阈值分 | 1 分即失败；比 0 大是为了让 0 有意义（0 表示必须完全相等）。调大等于小额差异被静默吞掉。 |
| 强自动匹配阈值 | 低于此值自动匹配会把不同项目并成一个，代价远大于人工复核。调低会误合并项目，调高会让人工队列暴涨。 |
| 人工复核阈值 | 0.70 到 0.85 是「像但不确定」，交给人判断。 |
| 拒绝阈值 | 低于 0.50 连候选都不算，避免垃圾候选污染队列。 |
| 允许浮点 | 浮点在累加万级行时必然产生分级误差，与零差异互斥。不要改——改了零差异在数学上不可能成立。 |
