## 三、报错了怎么办

| 遇到什么 | 为什么 | 怎么解决 |
|---|---|---|
| 渲染 FAIL：超行数上限 | 机器平面塞太多进人类平面 | 去精简机器平面，不要调大上限。 |
| 渲染 FAIL：中文门 | 正文出现未登记的英文术语 | 在机器平面 `glossary.json` 的 terms 加中文条目。 |
| 渲染 FAIL：功能清单出现日志词 | 又在往功能清单里写步骤日志 | 挪到 `05_执行与验收.md`（机器平面 runs）。 |
| 阻塞重审门 FAIL | 同一 Owner 阻塞被第 2 次重审 | 停。阻塞上浮 `00_我在哪.md`，本次结束。 |
| 报告等级恒为 D | 血缘不完整 | 查源优先级链第 1 环有没有接通。 |
| 零差异校验无法运行 | 无 A0 权威基准 | 解 BLK-001。 |
| 执行者仍从旧主仓运行或提交 KMFA | 仓库拆分后的工作目录、技能链接或自动化提示词没有同步 | 切到 `LinzeColin/KMOS`，读取 `HANDOFF.md`，验证稳定检出目录、技能链接和自动化工作目录。 |
| 原件摘要变了 | 原始数据被污染 | 立即停。从备份恢复原件，查是谁写的。 |
| 考勤、资金周报这些定时任务没动静，也收不到通知 | 2026-07-17 大清理删掉了旧工作树目录 `main_worktree/`，而 6 条在跑的排程工作目录全指向它，于是每天启动、找不到目录、失败。KMFA 已迁到本仓，旧路径不会再回来。 | 在 Codex 应用里把这 6 条排程的工作目录改到本仓在本机的实际位置。排程配在应用内部，改仓库没用。清单见 `KMFA/skills/registry.yaml` 的 `schedules` 段。 |
| 考勤或上游归档报「找不到 `dws`」、认证不通过 | 这两个技能靠外部命令 `dws`（钉钉工作台命令行工具）干活，Owner 本机没装，`~/.dws/` 也不存在。这跟上一条是两个各自独立的毛病，都得修。 | 装好 `dws` 并配认证；或用环境变量 `DWS_BIN` 指到实际可执行文件。它是开源的，不绑定任何一家客户端。 |
| 换台机器或搬上云，红圈相关的技能就失效 | `hongquan-main-contract-dws` 和 `info-fee-update` 各写死了 1 处 Owner 本机绝对路径。 | 把那两处改成配置项或环境变量，别写死。 |
| 找不到某个技能，或不知道哪条排程在用它 | 技能和排程长期分家：技能在仓库、排程在客户端，两边没有交叉引用，出事查不到。 | 查 `KMFA/skills/registry.yaml` —— 8 个技能、7 条排程的依赖关系都在里面。以后新增技能或排程，同步登记。 |
| changelog 新写了条目，双平面门全绿，但 06_运维手册 里看不到它 | 渲染器截断/切片类缺陷两侧一致（重渲染=现文件），渲染一致门必然绿——2026-07-18 曾因 changelog 新条目在前而渲染取 [-10:]（列表尾=最旧），第 11 条起最新条目静默隐身。 | 先跑 `KMDatabase/machine/tools/check_dual_plane_ci.py`（语义门会指名缺的条目）；渲染窗口约定=最新 10 条（changelog[:10]，新条目 `insert(0)`）。改渲染器切片必须带正负例验证。 |
| 收发明细表的「结存金额」列与 期初+收−发 对不上 | 源表固有显示伪象：物料清零段末行结存列不写 0（常写成当期收/发额），同日多单行的滚动余额按别的排序计算。两版工作簿逐分一致，非抽取失真。 | 结存列仅供参考；任何下游计算一律用收/发金额列自行滚动（审计见 `KMFA/stage_artifacts/DT5_DATA0035_gm_identity/`）。 |
| 排程一条都没执行，日志只有 `root: command not found` | `crontab.txt` 是系统格式（带 `root` 字段），被装成了用户 `crontab` | 只走 `/etc/cron.d`；`entrypoint.sh` 已加自检，装不上直接拒绝启动。 |
| 排程跑了但消息发不出去，`ledger` 记 `delivery_enabled` 为 `0` | `cron` 不继承容器环境，`cron.d` 也不做变量展开 | 部署时把 `KMFA_DELIVERY_ENABLED` 设成 `1`；`entrypoint.sh` 用实际值渲染进 `cron.d` 头部。 |
| 考勤首跑报 `DWS_AUTH_REQUIRED` | `cron` 环境没有 `HOME`，`dws` 找不到凭据目录 | `cron.d` 头部必须有 `HOME=/root`；已纳入 `entrypoint.sh` 自检。 |
| 派发结果 `FAILED` 但不说原因，目标列表为空 | 零目标时一路掉到汇总函数，被归成裸 `FAILED` | 已显式化为 `NOT_SENT_NO_TARGET_SELECTED` 等状态，凡未发出必带原因。 |
| 推了代码但服务器还是旧行为 | 部署工作流的 `Coolify` 接口令牌失效，返回 `401` | 见「五、服务启停」——换令牌后重部署；令牌只有 Owner 能生成。 |
| 考勤报告在容器重建后全丢 | 归档根写死成 `Mac` 路径，容器里只是个假目录 | 用 `KMFA_ATTENDANCE_ARCHIVE_ROOT` 指向持久卷（云端默认 `/var/lib/kmfa/attendance`）。 |
| 令牌失效，以为完全没法部署了 | 把「自动部署」和「能否部署」混为一谈 | `Coolify` 面板手动 `Redeploy` 照样能把 `main` 最新代码部上去；令牌只管自动化那条链。 |

## 四、服务启停

| 场景 | 怎么做 |
|---|---|
| 本机开 `App` | 双击 `KMFA.app`（自动起容器、等就绪、开页面） |
| 本机停 `App` | `docker rm -f kmfa-app-local` |
| 云端重部署 | **手动**：`Coolify` 面板进应用点 `Redeploy`，从 `main` 拉最新代码——**不需要 `API` 令牌**，令牌只管「推代码自动部署」那条自动化。**自动**：推 `main` 触发，前提是令牌有效 |
| 云端换接口令牌 | `Coolify` 生成新令牌 → 更新仓库 `COOLIFY_API_TOKEN` → 重部署 |
| 开启真实投递 | 部署时设 `KMFA_DELIVERY_ENABLED` 为 `1`，否则排程按空跑 |
| 在服务器上敲命令 | 不必 `SSH`——`Coolify` 面板自带 `Terminal`（`skills` 服务下），当初 `dws` 设备码登录就是在那儿做的 |

## 五、上线清单（顺序不能乱）

| 步 | 动作 | 不做会怎样 |
|---|---|---|
| 1 | `Coolify` 生成新 `API` 令牌 → 更新仓库 `COOLIFY_API_TOKEN`（**只影响自动部署**，手动重部署不需要它） | 推代码不会自动上线；仍可在面板手动 `Redeploy` 顶上 |
| 2 | `Coolify` 设 `COMPOSE_PROFILES` 为 `full` | `kmfa-app` 藏在 `full` 后面，容器根本不起，网站是空的 |
| 3 | `Coolify` 给 `app` 服务设域名 `kmfa.linzezhang.com` | 无 `Traefik` 路由 |
| 4 | **先做第 5 步**，再在 `Cloudflare` 加代理 A 记录 `kmfa` 指向主机公网地址 | 域名解析不到 |
| 5 | `Cloudflare Zero Trust` → `Access` → `Applications` 加自托管应用，策略只放行 Owner 邮箱 | **财务数据裸挂公网**——App 展示八份一致性报告与差异工作台，这是硬红线 |
| 6 | 部署时设 `KMFA_DELIVERY_ENABLED` 为 `1` | 排程按空跑，钉钉消息仍发不出去 |
| 7 | **手动**打开 `kmfa.linzezhang.com` 确认：先撞 `Access` 登录 → 通过后见页眉三元组 | 流水线**不会替你确认**——线上冒烟只在 `home_live_url` 非空时才跑，而 KMOS 的 `deploy.yml` 里它是空的；填上它又会连带触发 `home` 同步（缺 `HOME_SYNC_TOKEN` 会转红），故这一步只能人工。结果就是：部署显示绿色 ≠ 网站真能打开。 |

## 七、断链自检

一条链断了往往不报错，只是安静地不干活。定期照这张单子过一遍。

| 查什么 | 怎么查 |
|---|---|
| 排程真装上了吗 | 容器内 `grep -c run_skill /etc/cron.d/kmfa-skills`，应为 `10` |
| 环境行齐吗 | 同文件头部应有 `HOME`、`TZ`、`KMFA_DELIVERY_ENABLED` |
| 排程真跑过吗 | 看 `/var/log/kmfa/ledger.jsonl` 有没有排程来源的记录 |
| 认证还活着吗 | `dws-keepalive` 每四小时一次，看日志有无刷新失败 |
| 页面还能用吗 | 跑端到端回归 `app-e2e`，八步全绿才算通 |
| 部署链通吗 | 看仓库 `Actions` 里 `deploy` 是不是绿的；红且报 `401` 即令牌失效 |
| 状态面还是只追加吗 | 开库执行一次 `UPDATE`，应被触发器拒绝 |
| 网站真能打开吗 | 浏览器开 `kmfa.linzezhang.com`，应先撞 `Access` 登录、通过后见页眉三元组 |
