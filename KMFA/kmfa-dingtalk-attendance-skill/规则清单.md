## 1. 字段角色

- 钉钉企业每日统计原件包含 49 列。
- `部门`是唯一可选展示字段。有可靠主部门来源时可以显示；没有可靠主部门来源时记录 `UNVERIFIED`，不得猜值。
- `部门`不得参与考勤人员匹配、异常判断、月累计、通知内容或官方考勤验收，也不得单独阻止正式对账通过。
- 正式必需字段为其余 48 列：姓名、考勤组、工号、职位、UserId、日期、workDate、班次、三组上下班打卡时间与结果、关联的审批单、出勤/休息/工时/迟到/早退/缺卡/旷工/出差/外出统计、八类请假、加班总时长和三类转调休字段。

## 2. 人员与值语义

- 人员匹配只使用文本 UserId；必须保留前导零。
- 官方空白与数字 0 是不同值。只有经过独立官方原件验证的展示换算才能把来源零值显示为空白。
- 日期显示、Asia/Shanghai 日界毫秒、打卡时间和数字显示必须使用统一确定性换算。
- 审批后的最终打卡状态优先于旧运行时点状态；补卡审批结果不得由旧打卡快照覆盖。

## 3. 官方对账通过条件

单个工作日的正式凭证必须同时满足：

- 官方人数、DWS 人数、UserId 匹配人数相等且大于 0；
- 比较列数固定为 48；
- 比较单元格数等于匹配人数乘以 48；
- 缺少人员 0、多出人员 0；
- 必需列缺失 0、必需单元格缺失 0；
- 必需字段真实差异 0；
- 官方文件 SHA-256、工作日期和统一凭证 schema 完整有效。

测试通过、程序完成、人数汇总一致、DWS 内部 parity 或可选字段未验证均不能替代上述门槛。

## 4. 正式对账凭证

- `official_report_reconstruction.py` 负责生成统一的逐日正式凭证。
- `final_reconciliation.py` 直接验证和读取同一凭证 schema，不使用第二套转换格式。
- 凭证只保存聚合计数、官方文件指纹、工作日期和可选字段状态；不包含员工明细或报告正文。
- 2026-07-09 与 2026-07-10 的离线凭证均为 44/44/44 人、48 列、2,112 格、零必需缺失、零真实差异，`部门`为可选未验证字段。

## 5. 运行与发送边界

- R4 通过只证明冻结日期的官方考勤数据还原满足正式 48 列门槛。
- 真实 2026-07-10 evening 无发送运行及凭证绑定 final 已通过；owner usability 仍为 `NOT_ACCEPTED`，直到首次自然群发送留证。
- morning/evening 仅在实时完整性 PASS 后向现有 group target 发送；任何查询、漏查、日期或解析失败必须零 sender 调用。manual/latest-report resend 继续返回 `NOT_SENT_OWNER_DISABLED`。
- 晨间和晚间仍为暂时提醒；新月累计只读取 manifest 内绑定有效正式凭证的 canonical final archive。未绑定凭证的旧 final 同样只作审计。
- scheduler 按 owner 授权固定为本机 10:35 / 20:05，且无 timezone 字段；automation ID、cwd、发送目标和通知文案保持不变。

## 6. 证据与公开安全

- 官方 Excel 原件、员工数据、DWS raw 和逐格差异只保存在本机私有证据层，不提交 GitHub。
- GitHub 只保存代码、测试、规则、schema 和脱敏聚合结论。
- 官方原件、预期答案和通过条件在修复或复核阶段不得修改。

## 7. R6 自动闭环与幂等规则

- 通知预览不再是验收项；现有群通知模板及相关模板代码完全冻结，不得修改任何文字。
- latest pending work date 的独立官方 XLS/XLSX 必须由自然 automation 自动查找或使用现有钉钉登录态只读导出；不得要求 owner 人工搬运证据。
- 报告尚未生成时只记录 `WAITING_OFFICIAL_REPORT`，下一次自然 morning 自动继续，不得伪造通过。
- 原件必须先冻结 SHA-256，再生成统一 48 列正式凭证；凭证由自动协调器直接传给 `final_reconciliation.py`。
- 幂等键固定为“工作日期 + run slot”；final 与月累计只能从有效凭证绑定的 canonical final 写一次。
- 旧历史、morning、evening、未绑定凭证 final 均不得进入新的月累计。
- 不设置统一自然运行天数上线门槛；晨间自然成功、晚间自然成功、事后核验和发送状态分别记录，互不替代。
- 发送幂等键为“工作日期 + run slot”；外部调用前必须先写 `SEND_STARTED`，已有 `SENT` 或 `SEND_STARTED` 时返回 `NOT_SENT_DUPLICATE_GUARD`，不得再次调用 sender。
- 官方两行合并表头必须按二级非空优先、否则一级解析，且最终顺序严格等于 49 个 `OFFICIAL_COLUMNS`；请假和转调休分组标题不得拼接。
- morning 必须先完成当天 temporary reminder，再独立处理事后 final；final 错误不得阻断或改变 reminder。
- 自然证据必须来自真实 Codex automation task，记录 commit、两个 prompt 指纹、automation ID 和触发时间；CLI 自报不构成证据。
- artifact 恢复保留原始来源；attendance runtime 或 prompt 指纹变化只触发受影响项目复核，不删除既有事实记录、不全局清零。
- 官方人数 N 来自当天唯一非空 UserId，48 个必需字段固定，比较规模为 `N×48`。
- reminder 已成功而 final 为 `WAITING_OFFICIAL_REPORT` 时，总任务正常完成，显示“提醒成功，事后核验等待”；final 等待作为独立待办，不产生任务卡 system error。
