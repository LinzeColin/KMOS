## 固定运行序列

1. 晨间/晚间 automation 对当前考勤组逐人执行实时 `record get + summary`；全部查询且解析成功后落盘 `TEMPORARY_REMINDER` 并向既有 group target 发送一次。成功但无打卡记录仍是完整覆盖；失败零发送，不要求 final 官方报告状态。
2. 晨间 automation 保存当天 reminder 后，再独立查找最新 canonical evening reminder 且尚无 final 的已完成工作日，并按工作日期自动查找已下载的独立官方 XLS/XLSX。
3. 如本机尚无原件，automation 使用现有钉钉登录态只读导出完整官方日报；报告尚未生成时写入 `WAITING_OFFICIAL_REPORT`，不向 owner 索要文件，由下一次自然 morning 自动续接。
4. 取得原件后冻结指纹，调用现有 48 列还原器生成正式凭证，并由协调器直接传给 `final_reconciliation.py`；禁止人工拼接或提供 `--independent-reconciliation-evidence`。
5. 无独立原件时固定等待；任何缺人、多人、必需列/单元格缺失或必需值差异均固定 `OFFICIAL_FINAL_RECONCILIATION_BLOCKED`。
6. 只有统一正式凭证 PASS 且后续 DWS 内部门禁也通过，才可写 canonical `final` raw/manifest 和 aggregate-only `one_page.md`。
7. 新月累计只读取 manifest 内绑定有效正式凭证的最新 canonical `final` archive；未绑定凭证的旧 final、legacy 和暂时提醒均仅保留审计。
8. final 官方原件等待、导出或核验失败不得延迟、阻断或改变已经完成的 morning/evening reminder。
9. 私有协调器按工作日期与 run slot 去重，状态原子落盘；中断后先探测 artifact 完整性，final 与月累计只写一次。

- branch=`main` 且 `HEAD == origin/main`。
- DWS 授权和 browser policy READY；未授权则 fail closed。
- 实时完整性 PASS 后要求 `notification_status=SENT`；失败时 sender 调用数必须为 0。
- 不执行 target probe 或 resend；只允许 automatic runner 对现有 group target 做一次幂等发送。
- 不把 synthetic fixture 作为真实工作日证据。
- 私有 raw、报告正文、一页结果和 manifest 只留在私有归档，不提交 GitHub。
- GitHub 只保存脱敏聚合一页结果，不保存员工明细、官方原件、私有路径或逐格差异正文。
