## 2. Press Release 候选稿（Working Backwards）

> **用途说明：这是未来达到全部 GA Gate 后才可发布的候选稿，不是对当前生产的上线声明。**

### KMFA 让一项工作从资料到结果，不注册也能继续、带走和恢复

KMFA 面向想立即整理项目资料、记录进度并获得可解释经营结果的人。打开根域名即可开始；不要求
账号、邮箱或第三方登录。用户可以建立项目、上传资料、记录步骤与分数、查看处理结果，并下载原件
和报告。

今天，第一次访问 KMFA 的人会直接撞上登录墙；长期使用者还无法得到“换设备、清浏览器、重部署后
仍能取回同一份工作”的完整证明。文件格式限制、公开网站与私密资料的混淆，也会迫使用户在多个工具
之间拆散项目，或在不知道风险的情况下分享内容。

达到 GA 后，KMFA 将让首次使用者立即进入一个默认不公开的工作空间。资料和项目状态不会因为一次
会话或一次发布自动消失；用户保存好恢复材料后，可以在另一台设备继续。任何文件类型都可以进入
安全存储，但“可以保存”不等于“会执行、解压或预览”；原件始终可以按确定规则取回。公开分享必须
由用户明确发起，并且可以撤销。

KMFA 继承现有的财务核对、人工拍板、影响预览、重跑和报告下载主线，同时把结果来源、错误边界和
不可用状态讲清楚。它不会把未知金额补成看似完整的答案，也不会用注册或把用户文件公开来换取安全。

只有当匿名完整旅程、跨设备恢复、文件取回、私密泄露负向检查和回滚演练全部通过时，这份候选稿才
能改为上线公告。当前生产仍被 Access `302` 阻断，因此现在不能声称上述体验已经交付。

本稿没有客户引语：授权输入中没有可公开、可核验的真实访谈逐字稿。后续如加入引语，必须来自获得
同意的真实用户并可回溯，禁止编造“示例客户”来制造社会证明。

## 3. Customer FAQ

### FAQ-01｜这是为谁解决什么问题？

**答：**它首先服务于不想先注册就试用的访客、需要长期管理项目资料的个人，以及需要可解释经营结果
的分析者；核心问题是“能开始，但不能连续、取回或信任结果”时，工具没有完成客户工作。选择性分享者
和维护人员分别需要默认私密、可撤销发布，以及可验证的恢复与发布证据。

- 落点：`用户结果`
- 后续证明：端到端匿名旅程与真实行为信号；产品 Gate 参考 `AC-PUB-002`、`AC-QA-001`。
- 反证：若目标人群无法完成真实项目任务，或只把 KMFA 当一次性静态报告，则价值主张需要收窄并回到 Owner 决策。

### FAQ-02｜现在打开根域名就已经能完整使用吗？

**答：**不能。当前匿名访问仍被登录边界重定向，本文件只是客户合同候选，不是把失败状态改写成成功。
根入口直接显示完整主页、且全程没有账号步骤后，才跨过第一道可用性门。

- 落点：`风险`
- 后续证明：`AC-PUB-001`、`AC-PUB-002`、`AC-PUB-004` 的生产 Oracle。
- 反证：任何 `302` 登录跳转、账号字段、隐藏路径要求或首屏关键入口不可达，都直接否定“已公开可用”。

### FAQ-03｜为什么不要求账号？

**答：**客户可以立即验证产品能否解决手头工作，不必先交出邮箱、创建密码或绑定第三方身份；长期连续
使用通过由客户自己保存的恢复材料实现。免账号不等于无保护，也不能把登录重新包装成安全必需品。

- 落点：`用户结果`
- 后续证明：匿名核心旅程登录阻断数为 `0`，并验证正常用户在安全护栏下仍能完成任务；参考 `AC-PUB-002`、`AC-WS-004`。
- 反证：若恢复负担导致目标用户大规模无法继续工作，必须改进恢复体验或触发范围复议，不能偷偷加入强制登录。

### FAQ-04｜换设备或清理浏览器后，怎么继续同一个项目？

**答：**首次使用时，KMFA 会让客户保存一次恢复材料；在新设备导入后，应看到同一项目、进度、分数、
文件和可用结果。因为没有邮箱账户，丢失唯一恢复材料不能承诺人工找回，这一后果必须在客户离开前讲清楚。

- 落点：`风险`
- 后续证明：有效恢复数据一致率 `100%`、无效材料越权成功 `0`、秘密泄露 canary `0`；参考 `AC-WS-002`、`AC-WS-003`。
- 反证：标准恢复夹具有一次丢失、恢复提示不能被用户理解，或恢复材料进入 URL/日志/缓存，均否定该承诺。

### FAQ-05｜“长期保存直至删除”到底承诺什么？

**答：**客户没有明确删除时，项目状态和文件不因会话结束、应用重启或正常发布自动到期；备份恢复后也
必须仍可取回。它不是对物理灾难作绝对“永不丢失”保证，法律或安全处置例外必须有批准和审计。

- 落点：`用户结果`
- 后续证明：跨重启/部署记录丢失 `0`、对象与索引不可解释不一致 `0`、标准恢复演练 `100%`、误删 `0`；参考 `AC-DATA-001` 至 `AC-DATA-004`。
- 反证：只能在当前浏览器找到数据、备份无法从空环境恢复，或删除/保留行为没有确定规则，都否定“长期保存”。

### FAQ-06｜真的可以上传任意文件吗？

**答：**任意扩展名和媒体类型都可以被评估并安全保存；未知、可执行或高风险格式可以仅作为附件取回，
不会因此被运行、解压执行或直接嵌入页面。客户会看到真实大小、配额、处理状态和拒绝原因。

- 落点：`风险`
- 后续证明：标准合法文件可存、危险执行 `0`、恶意逃逸 `0`、合法夹具误拒低于批准阈值；参考 `AC-UP-001`、`AC-UP-003`。
- 反证：为了“支持全部格式”执行不可信内容，或以扩展名随意拒绝合法附件，均否定该承诺。

### FAQ-07｜网站公开后，我上传的资料会被别人或搜索引擎看到吗？

**答：**不会自动公开。新工作区、文件和结果默认不公开；只有客户明确选择发布，才生成受限且可撤销的
公开版本。公开网站、客户工作区和维护入口是不同边界，不能互相借道读取。

- 落点：`风险`
- 后续证明：默认公开命中 `0`、私密 canary 在页面/API/搜索/缓存/对象链接/遥测命中 `0`、撤销在 SLA 内完成；参考 `AC-PROD-003`、`AC-SEC-001`、`AC-ARCH-001`。
- 反证：未操作发布即可访问资料、公开快照夹带非白名单字段，或撤销后仍可命中，立即阻断发布。

### FAQ-08｜除存文件外，KMFA 能完成什么工作？

**答：**客户可以建立项目，记录步骤、进度、分数、备注和关联资料；经营/财务场景继续支持导入、核对、
解释、人工拍板、影响预览、重跑和报告。结果不确定时要显示阻断和来源，不能补造数字。

- 落点：`用户结果`
- 后续证明：项目 CRUD/恢复一致、财务 golden/black path、来源链和报告数值全部通过；参考 `AC-PROD-001`、`AC-FIN-001` 至 `AC-FIN-004`。
- 反证：用户只能浏览静态页面、变更无留痕、重跑静默覆盖、结果无法回到来源，均说明业务工作未完成。

### FAQ-09｜我能把自己的资料和结果完整带走吗？

**答：**客户应能下载原件、批准的派生物和报告，并看清文件名、类型、大小、来源与校验信息；读取和下载
本身不应暗中改变业务状态。高成本导出必须让客户明确创建、查看进度、取消或重试。

- 落点：`用户结果`
- 后续证明：可用制品下载率 `100%`、下载 hash/元数据不一致 `0`、读取副作用端点 `0`；参考 `AC-DL-001`、`AC-DL-003`、`AC-DL-004`。
- 反证：原件无法取回、重复下载改变状态、一次读取触发无预算长任务，均否定数据可携带性。

### FAQ-10｜免账号和任意文件是否意味着无限容量、无限处理或免费出网？

**答：**不意味着。客户会在写入或处理前看到可理解的容量、大小、频率与并发边界；正常使用不应被攻击
流量误伤，超限也不能造成数据悄悄丢失。限制必须来自实测能力和预算，而不是前端猜测。

- 落点：`非目标`
- 后续证明：攻击不造成无界增长、正常夹具误伤率低于批准阈值、无预算资源 `0`；参考 `AC-WS-004`、`AC-OPS-001`。
- 反证：成本只能靠隐藏限制控制、超限后数据状态不明，或正常旅程持续被拦截，说明公开模型不可运营。

### FAQ-11｜这轮会不会变成通用网盘、完整 ERP、BI 或协作平台？

**答：**不会。KMFA 只扩展完成核心客户旅程所需的项目、资料、可解释经营结果、恢复和取回能力；多人协作、
组织账号、计费、独立搜索集群、完整 ERP/BI 和为工具名迁移基础设施不进入当前核心交付。

- 落点：`非目标`
- 后续证明：每项新增范围必须追到当前客户结果与 Acceptance；没有收益证据的能力保持关闭或 backlog。
- 反证：若新增模块推迟匿名 walking skeleton、恢复或下载，或没有对应客户结果，它就是范围漂移，应从本轮移除。

### FAQ-12｜怎样知道它真的可以上线，而不只是演示可用？

**答：**只有真实生产身份可追溯、匿名核心旅程和负向安全检查通过、数据从备份恢复成功、渐进发布异常能
自动停止或回到上一可用版本时，才能宣布 GA。发布失败不会靠删除用户数据或重写历史来“回滚”。

- 落点：`风险`
- 后续证明：`AC-QA-001`、`AC-DATA-004`、`AC-REL-004` 及全部 P0 Gate；发布后 Oracle 必须绑定唯一制品和部署。
- 反证：只能提供文档、无法复现恢复/回滚、告警携带私密内容，或制品身份不唯一，都阻断上线声明。

| `FQ-02` | 根域名直接主页是否显著降低 dead end，还是用户仍找不到下一步？ | `/` 首访可用性测试、首个有意义动作完成率 | 重做信息架构与 onboarding；不以 `/ui` 隐藏路径规避 |
| `FQ-03` | 用户能否理解并保存恢复材料，以及“丢失无法邮箱找回”的后果？ | 两设备恢复测试、延迟回访、提示理解率 | 优先修复恢复 UX；标准夹具或理解门不通过则阻断 GA，不偷偷加登录 |

| `FQ-05` | “任意文件”真实分布是什么，哪些格式只需附件保存而无需预览/解析？ | 仅记录非敏感类型/大小/处理状态的隐私友好统计与夹具矩阵 | 保留任意类型安全存储；把低收益处理器降为 attachment-only，避免解析器膨胀 |
| `FQ-06` | 客户是否把 public site 错当成“上传即公开”？ | 文案理解测试、默认发布误操作率、私密 canary | 未达到理解与泄露门则不开放发布；改进状态与确认，不默认公开 |
| `FQ-07` | 显式公开分享是否有足够真实需求支撑 GA 内投入？ | 发布意图访谈、受控 beta 使用率、撤销任务成功率 | 若需求低，保留默认私密与撤销合同，把增强分享体验后置，不建公共内容平台 |
| `FQ-08` | 项目/进度/分数是否与现有财务业务流形成一个任务，还是两个无关产品？ | 端到端场景测试、任务切换和重复录入观察 | 只保留能减少重复或保持来源上下文的连接；若根本冲突，提交 Owner 做范围拆分决定 |
| `FQ-09` | 现有核对、拍板、重跑和报告流是否被真实使用且能迁移到匿名模型？ | 仓库 E2E + 合成业务夹具 + 经授权用户任务测试 | 保留已证明的业务不变量；删除纯展示/无人使用分支，任何真实数据迁移另过 Gate |
| `FQ-10` | 客户能否完整下载、校验并重新使用自己的原件和报告？ | Unicode/大文件/中断/批量标准夹具与任务测试 | 任一原件不可取回即阻断 GA；优先修下载，不用“可在线查看”替代可携带性 |
| `FQ-11` | 匿名限额能否同时控制成本和滥用，且正常用户误伤可接受？ | 正常/攻击负载、单位成本、误伤与降级测试 | 先收紧可见预算和高成本处理；若仍不可运营，P1.3 触发容量/范围决策，不改成强制账号 |
| `FQ-12` | 渐进发布和回滚是否在真实故障下恢复客户旅程，而非只切换版本号？ | 故障注入、前后数据不变量、核心旅程和唯一制品身份 | 回滚 Oracle 不通过则保持 previous artifact、Flag off，禁止 GA |
