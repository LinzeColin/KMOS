# ADR-0001：数据面与应用状态面分离，应用状态面用 SQLite

- **状态**：已采纳（2026-07-20）
- **任务**：TSK.KMFA.PROD.0001
- **决策依据**：D2=A（KMIDS 同栈：FastAPI + SQLite 应用状态 + React/Vite + ECharts + docker-compose）

## 背景

App 要读的东西和要写的东西，性质完全不同：

- **读**：`machine/facts`、`metadata/quality`、`machine/lineage.yaml` —— 这些是**治理数据面**，由数据线经审计流程生成，是全线的唯一真源。
- **写**：人在页面上做的决策、发起的重跑、导出的登记、留下的审计 —— 这些是**应用自己的状态**，与治理数据面不是一回事。

把两者混在一起写，后果很具体：App 一旦能写治理面，「事实」就不再是事实。

## 决策

**一、彻底分离两个面。** 数据面对 App **只读，永不写**；App 自己的状态落在独立的应用状态面。

**二、应用状态面用 SQLite**（`$KMFA_APP_STATE_DIR/kmfa_app_state.sqlite3`），五张表对应五类写入：

| 表 | 内容 | 来自 |
|---|---|---|
| `resolution_events` | 差异决策与冲正 | PROD.0007 |
| `rerun_steps` | 重跑四层链步骤 | PROD.0008 |
| `rerun_consistency` | 重跑一致性检查 | PROD.0008 |
| `export_records` | 导出 hash 登记 | PROD.0009 |
| `audit_events` | 审计事件 | PROD.0003 |

**三、append-only 由数据库强制，不靠约定。** 每张表挂 `BEFORE UPDATE` / `BEFORE DELETE` 触发器 `RAISE(ABORT)`。

## 为什么从 JSONL 换成 SQLite

0007–0009、0003 期间状态面先落成五个 JSONL，append-only 靠「我们只用 `"a"` 模式打开文件」这条约定维持。**约定不是保证**：一次手滑的写操作就能改掉一条已发生的事实，而且**不留任何痕迹**。

对一个记录「谁在什么时候做了什么决策」的系统，这条约定不够。SQLite 触发器把它变成库层强制——即便有人直接开库执行 `UPDATE`，也会被拒绝。

实测（宿主机直接开库，绕过 App）：
```
UPDATE audit_events …     → IntegrityError: audit_events 是 append-only，不得 UPDATE
DELETE FROM resolution_events … → IntegrityError: resolution_events 是 append-only，不得 DELETE
```

其余收益：`WAL` + `synchronous=FULL`（崩溃安全、写完真落盘）、单文件便于随实例备份、按序读天然有序。

## 代价与放弃的方案

- **代价**：状态面不再是能直接 `cat` 的纯文本。缓解——`/api/审计日志` 等接口照常吐 JSON，`app_state.read()` 一行取全表。
- **放弃 Postgres**：单用户本机应用，多一个要运维的服务不划算；D2 也已拍板 SQLite。
- **放弃继续用 JSONL**：见上，append-only 只能靠约定。
- **放弃写进治理数据面**：那会让 App 有能力篡改唯一真源。

## 迁移

`app_state.migrate_jsonl()` 可把既有 JSONL 搬进对应表，**幂等**（已迁过则跳过，不会翻倍）。云端首次部署会带着空库起，不需要人工干预。

## 后果

- 数据面只读这条铁律现在有测试盯着（`test_data_plane_stays_read_only_after_app_writes`：App 写完后 `assertions.jsonl` / `lineage.yaml` / `data_pipeline.json` 必须逐字节不变）。
- 备份对象从「五个文件」收敛成「一个库文件」，运维手册第五节已相应更新。
- 将来若要多用户/并发，SQLite 的写锁会成为瓶颈——届时再评估，**现在不预设**。
