## 五、工作纪律与实战模板（血泪换来的，照抄）

- **单元仪式**：真起服务→真开页面→真看真实数据→证据入 `stage_artifacts/<单元>/验收记录.md`
  （写实际命令与输出）→ `索引.md` 加行 → roadmap DT6 status 补记 → 双平面
  `python3 KMFA/machine/tools/check_dual_plane_ci.py --root . --require-projects` PASS
  → PR → 门禁 → squash 合并 → 五件收尾。禁止拿 `/healthz`/pytest 绿冒充页面完成。
- **门禁 v4 + 合并守卫**（每一步带网络重试；**收尾必须在确认 `MERGED` 之后**——
  曾因网络超时让分号链在未合并时删了远端分支，靠 GitHub 侧幂等侥幸未损）：
  1) `gh pr view $PR --json headRefName` 必须等于自己的分支（带重试，空值≠通过）；
  2) 轮询 `gh pr checks` 至无 `pending`，任何 `fail` 即止，且必须显式出现
     `dual-plane pass`（「没跑检查」不等于通过，别假设会跑几个门）；
  3) `gh pr merge --squash --delete-branch` 后**再查 `state=MERGED` 才许收尾**；
  4) 收尾五件：代码合、PR 关、worktree 收、分支删（本地+远端）、`git gc`
     （**禁 `--prune=now`**，历史上因此丢过 2467 个提交）。
- **验证命令**（本机 python 3.9，一律容器）：
  - 后端：`docker run --rm -v "$PWD":/w -w /w/KMFA/app/backend python:3.12-slim-bookworm sh -c "pip install -q -r requirements.txt pytest httpx; python -m pytest tests -q"`
  - 前端构建：`node:20-alpine` 里 `npm ci && npm run build`
  - 全流 E2E（回归基线，改 UI 必跑；抖动修复须三次全新容器）：起 `kmfa-app` 容器后在
    `mcr.microsoft.com/playwright/python:v1.49.0-jammy` 里执行
    `KMFA/taskpack/v1_4/html_uiux/kmfa_html_human_flow_audit.py --app-url <url> --out r.csv`
    （八步硬断言：页眉三徽章/真写决策/真跑四层链/真下载三格式；页签靠
    `role=tab` 且 `textContent` 严格相等——图标只能是无文本 `svg`）。
- **已知坑速查**：`ops.json` 的 `troubleshooting`（二十余条全来自真实事故）。
  另三条新近教训：查询参数名必须 ASCII（`h11` 拒收非 ASCII 请求行，浏览器转义
  只是侥幸）；「部署了但用户看不见」先查缓存头与根路径挂了什么，别急着怀疑部署；
