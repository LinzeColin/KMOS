# 自动部署前端新鲜度验收记录（2026-07-20）

## Owner 要求原文

「需要是自动部署啊 每次后端任意 commit、update、merge branch、merge pr 都需要确保前端是最新的」。

## 现状拆解（两个独立断点，别混为一谈）

1. **触发面没毛病**：`deploy` 工作流本就在 `main` 任意 push 上触发、无路径过滤——
   今天两次合并后它都真跑了（07:39/07:50 两次 run），红在 `Coolify` 令牌 `401`。
   换令牌是 Owner 网页动作（两分钟路径已写进运维手册「服务启停」），令牌不经任何中间人。
2. **真正的结构性漏洞在镜像构建**：原 `Dockerfile` 单阶段 `COPY .`，前端吃的是
   **仓里已提交的 `dist`**——谁改了 `src/` 忘了重建提交，服务器前端就是旧的。
   `app-e2e` 工作流曾为此在 runner 上先 `npm run build` 兜底（注释原话：
   「曾拿仓里旧 `dist` 打镜像而不自知」），但 `Coolify` 那条构建路径没有这层兜底，
   一直裸奔。

## 改了什么

- **`Dockerfile` 改多阶段**：`node:20-alpine` 阶段 `npm ci && npm run build`
  从当次提交源码现编前端，`COPY --from` 落位在 `COPY .` 之后覆盖仓里那份。
  从此 `Coolify`／CI／本地 `docker build` 三条路同一保证：**任意提交，镜像内前端
  必是当次源码产物**。仓里 `dist` 只服务本机 `.app` 双击场景。
- **`app-e2e` 工作流退役 runner 兜底**：删掉前置 `npm` 构建步——保证已移进
  `Dockerfile`，前端构建失败会红在 `docker build` 步，红的语义不变。
- **运维手册**：「云端重部署」写明前端新鲜度不用管；「换接口令牌」升级为
  两分钟三步路径（`Keys & Tokens` 建最小权限令牌 → 仓库网页更新
  `COOLIFY_API_TOKEN` → 之后合并即自动部署）。

## 实测（真构建、真毒化、真全流）

- **多阶段构建**：`docker build -f KMFA/app/backend/Dockerfile -t kmfa-app:fresh .`
  成功，构建日志可见 `frontend` 阶段逐层执行（`npm ci` → `vite build`）。

  镜像内现建资产 `assets/index-D2Pemf60.js` / `assets/index-C3OByXGn.css`——
  **镜像对仓内 `dist` 零依赖**。实验后 `dist` 原样放回，工作树恢复干净。
- **全流端到端**：CI 同版本 playwright 1.49 容器对 `kmfa-app:fresh` 跑八步
  真流程，**10/10 PASS**（真写决策、真跑四层链、真下载三格式）。

## 不含糊的边界

- 本单元**修不了** `401` 本身——新令牌只有 Owner 能在 `Coolify` 生成、只应经
  GitHub 网页粘进 secret。换完后「推送即部署+前端必最新」全链自动。
- 在令牌修好前，`deploy` 工作流每次合并仍会红（如实），手动 `Redeploy` 不受影响。
- `.app` 本机双击场景仍读仓里 `dist`——改前端后想让 `.app` 也见到新界面，
  需重建并提交 `dist`（服务器不受此约束）。
