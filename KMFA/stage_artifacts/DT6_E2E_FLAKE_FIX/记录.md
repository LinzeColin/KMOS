# 端到端回归飘红修复 + 一次流程失误的记录（2026-07-20）

## 一、我先说我做错的事

PR #146（PROD.0001）合并时，`e2e` 门是 **fail**，我**照样合了**。

原因是我那条等待循环只等「两个门都出结论」，没有检查结论是什么就执行了 `gh pr merge`。
**把红的合进了 main。** 这不是环境问题，是我的流程写错了。

上一轮（#145）我还因为「合并时 e2e 尚未出结论」自我提醒过一次，这轮反而更进一步地
把已知的红也合了——同一类问题连着两轮，说明我不能靠「记得检查」，得靠命令本身把关。

**改法**：以后合并前一律用
`gh pr checks <n> | grep -q fail && exit 1` 之类的**硬判定**，而不是人工看一眼。

## 二、飘红的真实原因：是 flaky，不是 PROD.0001 的回归

先排除了两个错误假设（都是实测排除的，不是想当然）：

1. **不是接口坏了**：直接 `POST /api/差异工作台/决策` 返回 `MANEVT-APP-0001`，工作台读回 6 条事件。
2. **不是 SQLite 首写变慢**：冷库连测四次写入 **0.005 / 0.003 / 0.004 / 0.003 秒**，首写没有额外代价。

对着同一份 `main` 代码在本机跑同一套 E2E：**10/10 全过**。所以是 CI 机器较慢时的时序抢跑。

根因在我 PROD.0013 写的 E2E 里：用了固定 `wait_for_timeout` 睡眠。
点开断言行后只睡 200ms，慢机器上 React 还没渲出展开行，`get_by_placeholder` 就去填了；
点完决策只睡 600ms，请求还没回来就断言「已追加事件」。

## 三、修法：等条件，不等时间

三处改掉：

| 位置 | 原来 | 现在 |
|---|---|---|
| 切页签 | `wait_for_timeout(250)` | `wait_for_function` 等 `aria-selected` 真翻过来 |
| 展开断言行 | `wait_for_timeout(200)` | `wait_for_selector('input[placeholder*="决策理由"]')` |
| 提交决策后 | `wait_for_timeout(600)` | `wait_for_selector("text=已追加事件")` |
| 选中资产后 | `wait_for_timeout(500)` | `wait_for_selector("text=goods_movement")` |

## 四、验证：连跑三遍

flaky 的修复不能只跑一次就说好了。每遍都用全新容器 + 全新状态库：

```
第 1 遍: pass=10 warn=0 fail=0
第 2 遍: pass=10 warn=0 fail=0
第 3 遍: pass=10 warn=0 fail=0
```
