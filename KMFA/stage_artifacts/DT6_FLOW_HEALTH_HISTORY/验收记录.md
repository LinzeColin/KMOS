# 流健康全历史验收记录（2026-07-21 凌晨）

## Owner 指令原文

「所有的skills都需要整合进kmfa功能模块的，但是也的确需要呈现线程健康（每个
workflow, skills flow, 操作流，分析流，数据流，工作流的顺畅度、是否运行、历史
运行记录等），目前你只有最后一次运行记录，但应该保存所有的运行历史记录和当时的快照」

## 事实澄清（先纠正一个认知）

**历史和快照在服务器上一直都在**：`run_skill.sh` 每次运行 append 一行进
append-only 台账，且**每次运行写独立时间戳日志** `/var/log/kmfa/<技能>/<时间戳>.log`
（即当时快照）。缺陷在 App 一侧——接口只取每技能最新一条、快照从未暴露。
本单元零改动运行侧，纯 App 侧补齐。

## 改了什么

1. **后端 `/api/排程健康`**：每技能新增 `业务模块`（考勤与日检/资金与经营报告/
   数据接入/系统底座——skills 归进功能模块）、`次数/失败次数/成功率/连续失败`、
   `历史[]`（全量最近在前，封顶 100 条，每条带当次快照路径）。
2. **后端新增 `/api/排程健康/快照?log=`**：读某一次运行的日志快照；路径防穿越
   （resolve 后必须在排程日志根内）；只回尾部 64KB。
3. **前端系统自检**：排程表按业务模块分组；新增 次数/成功率 列；点行展开全量
   运行历史；每条历史「看快照」当场内嵌显示当次日志。
4. **其他流总览**：操作流（拍板/下载留痕 N 条）/重算流（N 轮四层链）/数据流
   （批次），各自点击直达数据底账对应节——四类流的健康与历史一页收口。

## 实测（真起容器、真发请求、真开页面）

- **pytest 86 绿**（新增 2 条：全量历史统计断言；快照 200+目录外 404+相对穿越 404）。
- **容器夹具**（5 次运行：attendance-morning 三天含一次失败、dws-keepalive 两次）：
  接口返回 `次数3/失败1/成功率67%/模块考勤与日检`，历史按时间倒序；
  快照接口读出当次日志尾行 `结束 rc=0`；`/etc/passwd` 与相对穿越均 404。
- **浏览器实测**（截图）：系统自检页四个业务模块分组齐；attendance-morning
  行展开 3 条历史；点「看快照」内嵌显示 `/var/log/kmfa/attendance-morning/
  20260720_184800.log｜154 字节` 与日志全文。
- **全流 E2E 三次全新容器 11/11 全绿**。
- 过程中修一只真兼容问题：快照查询参数原用中文名 `路径`，`h11` 拒收非 ASCII
  请求行（浏览器自动转义故前端侥幸可用）——改 ASCII `log` 与既有 `?asset=` 惯例
  一致，curl/规范双通过。

## 不含糊的边界

- 服务器生效待合并后自动部署；历史从**已有台账**立即可见（台账自 #137 起全量），
  快照可见范围=志卷里实际存在的日志文件。
- 快照展示尾部 64KB 封顶——复盘用，不是日志下载器。
- 「分析流/工作流」中属 App 侧的操作流与重算流本就有各自 append-only 历史，
  本单元以总览卡收口入口；跨会话开发流（CI/PR）健康在 GitHub Actions 一侧，
  不重复搬进 App。
