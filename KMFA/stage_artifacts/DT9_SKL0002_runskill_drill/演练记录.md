# run_skill.sh 管道实弹演练记录（2026-07-18 容器内）

预检（`DT9_SKL0002_deploy_precheck`）验证了工具链；本演练验证 **cron 实际会走的那条管道**：`flock` 防重入 → 技能执行 → 台账 append → 告警分层。容器 = 预检同镜像（arm64），仓库只读挂载，无 secrets（按设计空跑语义）。

## 五场结果

| 场 | 内容 | 期望 | 实测 |
|---|---|---|---|
| 1 | `fund-weekly`（验证器） | rc=0 + 台账行 | ✅ `PASS: taskpack static validation`，台账 JSON 行完整（`delivery_enabled:"0"`） |
| 2 | `work-check-morning`，输入 zip 缺失 | 见下方定性 | ✅ rc=0 + 结构化 `SOURCE_MISSING` 事件 |
| 3 | 防重入（先占 `mgmt-monthly` 锁再入） | 第二发跳过 | ✅ 日志「上一轮仍在运行，跳过」，rc=0 |
| 4 | 未知技能名 | rc=2 | ✅ `未知技能: no-such-skill`，rc=2 |
| 5 | 台账/日志完整性 | 行结构完整 | ✅ `ledger.jsonl` 每次运行一行（ts/skill/rc/log/delivery_enabled） |

## 场 2 定性：分层告警设计，非缺陷

输入 zip 缺失时，`daily_routine_check` 把它作为**一等监控事件**处理而不是崩溃：

- 输出 `SOURCE_MISSING/ZIP_INPUT_MISSING` 数据质量条目，受影响规则进 `rules_blocked_by_source`（两条现金类规则），其余按窗口正常评估/跳过；
- 生成 `notification_events`（幂等键 `SOURCE_MISSING:付款请示群:<日期>:ZIP_INPUT_MISSING`），**通知对象=张霖泽**；双跑纪律下 `notification_delivery_status=LOG_ONLY_DRY_RUN`，开闸后走钉钉真发；
- rc=0 语义 =「检查本身成功并上报了源缺失」。

**告警两层由此分工**：技能内事件（源缺失/规则命中→通知张霖泽）｜`run_skill.sh` webhook（进程崩溃 rc≠0→运维告警）。生产上两层都活。

## 附带确认

- 锁跳过的运行不写台账（只写日志）——刻意保持台账=真实执行记录；跳过频繁时看技能日志。
- 工具内部时区自钉 `Asia/Shanghai`（check_date 按北京时间），与容器 TZ（悉尼，cron 排程用）互不干扰——两个时区各司其职，此为设计。
- `case` 表 8 条技能路径在中文重命名后全部有效（逐一 `-f` 核对）。

## 结论

管道全链路（锁/执行/台账/双告警/双跑旗标）在目标架构容器内实测通过，**无需改码**。实例日 cron 触发后即为本演练同路径。
