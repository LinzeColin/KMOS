# G5 一键签核通道验收记录（2026-07-20）

## 为什么有这个单元

任务包写死 `Approver: Owner`，但**签核一直没有入口**——Owner 想签也没处签，
G5 就永远停在 `g5_passed: false`。同时 Owner 已定纪律「本机仅为代码开发，
所有运行都在云端」，所以签核入口不能是本地脚本。

## 设计：签名住在台账里，检查器只认账不造账

- **签名落痕处**：`metadata/approvals/control_events.jsonl`（既有 append-only 审批台账），
  新增 `g5_signoff` 事件类型，字段沿用 `kmfa.control_event.v1`。
- **签核入口**：GitHub Actions `sign-g5`（`workflow_dispatch`）。Owner 网页上点
  Run workflow → 逐字输入确认语「同意签核G5」→ 可勾「已亲验网站」。
  **GitHub 登录态就是身份验证**——`github.actor` 必须是 `LinzeColin`。
- **三层防线**（`tools/sign_g5.py`）：签核人不对拒签（rc=3）；确认语不逐字拒签（rc=4）；
  技术判据现场重跑不全绿拒签（rc=5）——签名不豁免技术判据。幂等：已签不重签。
- **检查器改动**（`tools/check_g5_exit.py`）：`g5_passed` 由「恒 false」改为
  「识别台账中字段齐全的 Owner 签核事件」——工具链自始至终不制造签名。
- 签核成功后：刷新 `g5_checklist.json`（带签核记录）、roadmap DT6 标 `completed`
  （原状态全文留档）、跑双平面校验、由 workflow 提交回 `main`。

## 实测（草稿区完整拷贝上真跑六条路径，逐条输出）

```
a) 未签名：checker rc=0，tech_green: True | g5_passed: False（待办首条指向 sign-g5）
b) 确认语错：  FAIL rc=4「确认语必须逐字…签名是个动作，不是个默认值」
c) 签核人错：  FAIL rc=3「签核人必须是 LinzeColin…不代签，不例外」
d) 正确签核：  rc=0 事件 CTRL-KMFA-20260720-G5-SIGNOFF 落台账，
              g5_passed 翻 true，roadmap DT6 标 completed（原状态留档）
e) 幂等重跑：  「已签核过…不重复落痕」rc=0
f) 复核：      检查单含签核记录（事件号/签核人/时间/运行/亲验），台账尾行即该事件
```

测试在**一次性拷贝**上进行，测试用的模拟签名未进仓——仓内 `g5_checklist.json`
已用新检查器刷新，如实仍为 `g5_passed: false`（待 Owner 真签）。

## 不含糊的边界

- 本单元交付的是**通道**，不是签名。`g5_passed` 在仓里依然是 false，
  翻真只能由 Owner 在 GitHub Actions 亲手触发——这正是任务包要的权力边界。
- 勾「已亲验网站」是 Owner 的**亲证陈述**并落痕（完成判据第 5 条的记录面）；
  工具无法也不应替 Owner 看页面。
- 检查器旧待办里「换令牌上线」一条已过时（站点已上线），本次一并更正为
  「登录亲验」。
