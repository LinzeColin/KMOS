# 排程从未执行的真 bug — 修复记录（2026-07-20）

基建线（OVH Singapore VPS-1）报：**排程任务从上线以来一次都没成功执行过**，与投递开关无关。
Owner 说没收到钉钉，根因在我写的 `KMFA/deploy/skills-runtime/entrypoint.sh`。

## 一、我自己复现确认（没有只凭转述改生产代码）

原第 32 行：

```bash
crontab /opt/runtime/crontab.txt 2>/dev/null || cp /opt/runtime/crontab.txt /etc/cron.d/kmfa-skills
```

设计意图是「crontab 对系统格式文件会报错 → 走 `||` 右边写 cron.d」。**实测这个前提是错的**：

```
$ docker run --rm -v crontab.txt:/tmp/crontab.txt debian:bookworm-slim sh -c 'crontab /tmp/crontab.txt; echo $?'
crontab 退出码: 0            ← || 右边永不执行
$ crontab -l | head -1
35 10 * * *   root /opt/runtime/run_skill.sh attendance-morning  >> ...
$ ls /etc/cron.d/
e2scrub_all                  ← 没有 kmfa
第6字段(命令起点) = root      ← root 被当成命令
```

`crontab.txt` 是**系统 crontab 格式**（10 条全带 `root` user 字段），只对 `/etc/cron.d/` 有效。
`crontab <file>` 面对它不校验格式、不报错，把 `root /opt/runtime/run_skill.sh …` 整体当命令收下。
于是排程被静默装成**用户 crontab**，每次触发只吐 `root: command not found`。

**教训：别拿命令的返回码去猜它是否理解了输入格式。**

## 二、连带影响：#123 的保活修复其实从未生效

`20 */4 * * *  root … dws-keepalive` 就在这 10 条里。那是我在 #123「补回 dws 认证保活」时加的——
**因为这个 entrypoint bug，它一次都没跑过**，等于修了个寂寞。dws 刷新令牌自上线以来无人续期。

## 三、修法：只走 cron.d，且装不上就不许启动

```bash
CRON_D=/etc/cron.d/kmfa-skills
crontab -r 2>/dev/null || true   # 清残留用户 crontab，杜绝两种格式并存重复触发
install -m 0644 -o root -g root /opt/runtime/crontab.txt "$CRON_D"
if [ -n "$(tail -c1 "$CRON_D")" ]; then echo >> "$CRON_D"; fi   # cron.d 要求末尾换行
EXPECT_JOBS="$(grep -cE '^[0-9*].*run_skill' /opt/runtime/crontab.txt || true)"
GOT_JOBS="$(grep -cE '^[0-9*].*run_skill' "$CRON_D" 2>/dev/null || true)"
… 数目不符或为 0 → 打印理由并 exit 1
… 仍存在用户 crontab → 打印理由并 exit 1
```

自检的理由与既有 `+0800` 时区守卫一致：**静默失效比起不来危险得多**——上一版就是这样安安静静一个月不发消息，还得靠 Owner 说「我没收到」才被发现。

## 四、我的第一版自检自己也犯了同一类错

负例测试时容器确实退出了（码 1），但**日志里没有拒绝理由**。查出来是：脚本开了 `set -euo pipefail`，
而 `grep -c` 数到 0 条时**退出码是 1**，于是

```bash
EXPECT_JOBS="$(grep -cE '^[0-9*].*run_skill' /opt/runtime/crontab.txt)"
```

在这一行就把脚本静默打死，我写的拒绝理由根本没机会打印——**等于把「静默失败」又造了一遍**。
加 `|| true` 后才真正做到大声失败。

（同理把 `[ -n "$(tail -c1 …)" ] && echo >> …` 改成显式 `if`，避免 `set -e` 下这类 `&&` 惯用法的歧义。）

