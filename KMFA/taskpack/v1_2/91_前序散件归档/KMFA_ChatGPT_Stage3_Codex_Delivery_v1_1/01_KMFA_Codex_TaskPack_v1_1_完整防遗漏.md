# KMFA 经营分析系统｜ChatGPT 阶段三交付｜Codex TaskPack v1.1

## 0. 文件定位

当前对话处于 **ChatGPT 阶段三**。本阶段的交付物不是代码实现，而是给 Codex 使用的完整开发任务包与独立开发路线图。

本 TaskPack 的用途：

1. 把阶段一、阶段二五个环节、用户补充要求、上传材料、业务线模块、数据治理规则、公式参数、UIUX 规则、验收门禁全部汇总。
2. 防止 Codex 开发时遗漏关键要求。
3. 防止 Codex 被时间参考误导：时间只是参考，不得影响质量；完成得早可以提前交付，质量未过不得交付。
4. 防止 Codex 先冲 UI、先写报告、先接自动接口而绕过数据治理内核。

## 2. P0 目标

第一轮开发的 P0 目标是文件型项目成本分析 MVP：

```text
上传/登记权威项目成本资料
→ 原始文件hash与manifest
→ 字段映射
→ A0黄金基准
→ 金额整数分标准化
→ 零差异校验
→ 差异队列
→ 人工处理事件
→ 重跑派生链路
→ 项目成本报告
→ 经营总览摘要
```

P0 目标不是完整财务软件，不是报税系统，不是付款系统，不是工资发放系统。

## 6. 数据优先级与引用一致性

数据优先级固定为：

```text
原始上传数据 / 授权导出数据
> 原始抽取值
> staging结构化行
> canonical fact
> 派生指标
> 报告引用
> 前端展示
```

一致性规则：

1. 同一个原始数据被不同板块引用时，必须引用同一个source_id、file_hash、field_path、mapping_version。
2. 如果板块A与板块B引用结果不一致，必须失效派生缓存并重跑导入、映射、事实层、指标层、报告引用层。
3. 不允许在前端或报告层直接改数。
4. 不允许为展示效果改变事实层数值。
5. 重跑结果仍不一致时进入差异队列。

## 7. 项目唯一键与组合匹配

项目唯一键采用组合证据模型，不因缺失某单个字段就废弃数据。

| 字段 | 权重建议 | 说明 |
|---|---:|---|
| 合同编号 | 0.20 | 强主键，但可能缺失或不同模板写法不一 |
| 项目名称 | 0.18 | 强匹配，但需要标准化简称、错别字、括号 |
| 对手方/甲方/客户 | 0.15 | 匹配客户和项目归属 |
| 公司主体 | 0.10 | 多主体必需 |
| 产生时间/开工/完工/报告期 | 0.12 | 用于区分同名项目和跨期数据 |
| 合同额/开票额/结算额 | 0.12 | 金额辅助匹配，不可单独作为主键 |
| 项目经理/责任人 | 0.06 | 辅助匹配 |
| 来源文件hash/上传包 | 0.07 | 版本和来源区分 |

匹配等级：

```text
>= 0.85：自动强匹配
0.70 - 0.85：候选匹配，需人工复核
0.50 - 0.70：弱匹配，仅进入候选池
< 0.50：不匹配
```

## 8. 金额精度规则

1. 业务金额必须以整数分或Decimal计算。
2. 任何金额比较都以分为单位。
3. 0.01元差异即失败。
4. 空白、横杠、井号、异常文本不得默认为0。
5. 元、万元、千元、含税、不含税必须有字段或规则说明。
6. Excel公式值、显示值、PDF显示值出现冲突时，全部保留源值，进入差异队列，不自动取舍。

## 9. 口径转换与差异核对机制

本机制只作为系统内部数据治理与管理分析机制，不作为报告大标题。

必须支持以下核对：

```text
合同金额 ↔ 项目收入
项目成本 ↔ 财务凭证/费用明细
银行回款 ↔ 应收账龄
开票金额 ↔ 合同/结算/税务数据
研发费用 ↔ 项目/人员/成果证据
权威项目成本PDF ↔ Excel汇总 ↔ 系统复算
```

每条差异必须有：

```text
difference_id
source_a
source_b
field_name
amount_a_cents
amount_b_cents
delta_cents
reason_candidate
impact_scope
resolution_status
reviewer
created_at
closed_at
```

## 10. 数据质量与报告等级

### 10.1 数据质量 Q0-Q5

| 等级 | 含义 | 是否可生成正式内部报告 |
|---|---|---|
| Q0 | 未导入或无法读取 | 否 |
| Q1 | 已登记但未解析 | 否 |
| Q2 | 已解析但字段不完整 | 否 |
| Q3 | 机器候选结构化 | 仅可预览 |
| Q4 | 人工确认黄金字段 | 可用于内部复核报告 |
| Q5 | 零差异验证通过并完成追溯 | 可用于正式内部报告 |

### 10.2 报告等级 A/B/C/D

| 等级 | 条件 |
|---|---|
| A | Q5数据、零差异通过、关键差异关闭、人工确认完成 |
| B | Q4数据、关键差异可解释、报告明确限制 |
| C | Q3数据，仅供预览和复核 |
| D | 关键数据缺失、过期或差异未处理，不得作为经营决策依据 |

## 11. 前端和报告规则

1. 全中文。
2. KM标识，不用单K。
3. 蓝色系商务风，避免大面积黄色；警示只使用小型徽标或标签。
4. 首页模块为：经营总览、项目成本、回款应收、财务资金、开票纳税、数据源检查、待处理事项、报告中心。
5. 报告用语必须像真实经营管理软件，避免工具化、噱头化、过度技术化。
6. 报告中不得出现内部技术机制英文标题。
7. 数据源检查板属于系统功能，不塞进经营报告正文。
8. 报告只展示必要的数据状态、口径说明和待处理事项。

## 14. 完成定义

Codex 只有在以下条件全部满足时，才能宣布阶段任务完成：

1. 对应Stage/Phase/Task的验收证据存在。
2. 金额精度测试通过。

6. 前端操作不会直接污染raw层。
7. 报告符合全中文、人类可读、商务蓝、信息密度高的要求。
8. README、功能清单、开发记录、模型参数文件同步更新。
9. no_omission检查通过。
10. Go/No-Go评审通过。
