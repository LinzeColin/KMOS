# KMFA 经营分析系统｜数据关联模型 v0.1

状态：ChatGPT 阶段一讨论稿  
用途：说明系统如何把红圈、金蝶、WPS、银行、税务、合同和政策材料关联成经营分析结果，避免报告只有表面展示。

---

## 1. 核心结论

KMFA 的核心不是“多导入几个表”，而是建立一套项目制工程企业的经营关联模型：

```text
客户 → 商机/报价 → 合同 → 项目 → 成本 → 结算 → 开票 → 回款 → 税务政策 → 管理分析报告
```

系统内部按以下原则工作：

1. **唯一账本为基础**：金蝶等财务系统提供合法账本与财务事实基础。
2. **项目数据做经营解释**：红圈和 WPS 补充项目、合同、回款、成本和现场信息。
3. **银行流水验证现金事实**：现金安全、回款核销和付款压力必须与银行数据关联。
4. **发票税务提供政策口径**：税率、销项、进项、研发费用、高新与科小材料必须有证据链。
5. **报告只展示经营结论**：内部核对流程不在经营报告中展开，只保留必要的“口径差异摘要”。

## 2. 标准实体模型

| 实体 | 中文名称 | 主键建议 | 主要来源 | 关键字段 |
|---|---|---|---|---|
| CompanyEntity | 公司主体 | company_entity_id | 金蝶、银行、税务 | 主体名称、税号、银行账户、账套 |
| Customer | 客户 | customer_id | 红圈、WPS、金蝶 | 客户名称、账期、信用等级、联系人 |
| Opportunity | 商机/询价 | opportunity_id | 红圈、WPS | 来源、项目类型、预计金额、状态 |
| QuoteBid | 报价/投标 | quote_id | WPS、红圈 | 报价金额、预计成本、毛利、结果 |
| Contract | 合同 | contract_id | 红圈、合同目录、WPS | 合同额、税率、付款节点、质保金 |
| Project | 项目 | project_id | 红圈、WPS | 项目类型、负责人、进度、状态 |
| CostItem | 成本项 | cost_item_id | 金蝶、红圈、WPS | 成本类别、金额、期间、项目归属 |

| PolicyEvidence | 政策证据 | evidence_id | WPS、资料目录、税务材料 | 政策类型、项目、人员、费用、IP |
| ManagementAdjustment | 管理调整 | adjustment_id | 系统内部 | 差异来源、原因、金额、负责人、状态 |

## 3. 主关联键

| 关联对象 | 首选关联键 | 备用关联键 | 人工复核触发条件 |
|---|---|---|---|
| 客户 ↔ 合同 | customer_id | 客户名称规范化 + 税号 | 名称相似但税号不同 |
| 合同 ↔ 项目 | contract_id / project_id | 合同编号 + 项目名称 | 一合同多项目或多合同一项目 |
| 项目 ↔ 成本 | project_id | 摘要关键词 + 辅助核算 + 期间 | 成本无项目编码 |
| 项目 ↔ 发票 | project_id / contract_id | 客户 + 金额 + 期间 | 发票对应多个项目 |
| 发票 ↔ 银行回款 | invoice_id / customer_id | 对方户名 + 金额 + 日期窗口 | 金额拆分、合并付款、代付 |
| 应收 ↔ 回款 | receivable_id | 客户 + 合同节点 + 银行入账 | 银行已入账但应收未核销 |
| 研发项目 ↔ 费用 | rd_project_id | 项目编码 + 人员工时 + 摘要 | 研发/非研发边界不清 |

## 4. 关键经营链路

### 4.1 现金安全链路

```text
银行余额
+ 合同付款节点中的预计回款 × 回款概率
+ 已开票未回款 × 回款概率
- 已承诺付款
- 未来固定支出
= 预测现金余额
```

必要来源：

- 银行流水：主体、银行、账户、交易、余额
- 金蝶：应收、应付、科目余额
- 红圈：回款计划、项目节点
- 合同：付款节点、质保金、结算条件
- WPS：账期与催收状态

阻塞规则：

- 关键主体银行流水缺失：现金安全报告降级。
- 合同付款节点缺失：未来回款预测降级。
- 应收账龄缺失：回款优先级无法排序。

---

### 4.2 项目真实利润链路

```text
合同收入
- 人工
- 材料
- 机械
- 外协
- 运输吊装
- 加工
- 差旅
- 安全环保
- 税费
- 质保风险
± 签证变更
± 跨期成本管理归集
= 项目真实利润
```

必要来源：

- 红圈：项目、人材机、外协、进度
- 金蝶：凭证、科目余额、辅助核算
- WPS：报价、临时成本、项目台账
- 合同：合同金额、变更签证、结算
- 税务/数电票：进项、销项、税率

阻塞规则：

- 成本无项目归属：真实毛利只能输出区间，不输出确定值。
- 未确认签证：只能列为潜在收益，不进入确定性利润。
- 外协成本跨期：进入口径差异摘要，并触发人工复核。

---

### 4.3 订单与客户质量链路

```text
客户历史合作
+ 报价转合同率
+ 项目毛利
+ 回款速度
+ 争议频率
+ 复购潜力
= 客户质量评分
```

必要来源：

- 红圈：客户、商机、合同、项目
- WPS：客户信息、报价投标、账期
- 金蝶：应收账龄、客户往来
- 银行：客户实际回款

说明：

KMFA 不采用互联网广告投放模型，不把营销费用作为必填项。对项目制工程企业，商务质量主要来自客户质量、报价纪律、合同条款、项目利润和回款质量。

---

### 4.4 税务政策链路

```text
合同税率
+ 发票税率
+ 项目服务类型
+ 金蝶科目
+ 进销项匹配
+ 研发项目证据
+ 科技人员/知识产权/高新收入证据
= 税务政策准备度与风险提示
```

必要来源：

- 税务/数电票：销项、进项、税率、认证抵扣状态
- 金蝶：科目余额、凭证、费用归集
- WPS：发票台账、研发项目、科技人员、知识产权
- 合同：服务类型、税率、付款/开票条款

阻塞规则：

- 发票与合同税率不一致：进入税务政策人工复核。
- 研发费用没有项目与人员证据：只输出缺口清单，不输出政策资格肯定结论。
- 高新收入缺合同/发票/科目映射：不计算确定占比。

## 5. 报告层与系统层的边界

| 内容 | 系统内部需要 | 报告中呈现 |
|---|---|---|
| 数据源检查板 | 完整呈现 | 不进入正式经营报告，只在系统里查看 |
| 口径转换流程 | 完整执行并记录证据 | 只显示影响经营判断的差异摘要 |
| 数据质量校验 | 每批次执行 | 报告只显示数据状态与限制说明 |
| 参数阈值 | GitHub 文件中可审计 | 报告显示参数版本号 |
| 证据索引 | metadata 中保存 | 报告附录显示来源范围，不暴露敏感明细 |
| 人工复核 | 系统工作台处理 | 报告显示需要经营层处理的事项 |

## 6. 长期维护要求

1. 每个来源系统必须有 source registry。
2. 每个文件包必须有 schema map。
3. 每次导入必须有 import run。
4. 每个关键指标必须有 metric lineage。
5. 每份报告必须有 report manifest。
6. 每条人工调整必须有 owner、reason、evidence、status。
7. 每次参数变更必须有版本号、变更原因和回滚方式。
8. 后续接入 OpMe 时，只接入口、报告索引和运行状态，不混用敏感数据。
